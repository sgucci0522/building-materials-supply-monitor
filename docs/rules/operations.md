# 運用 / トラブルシューティング

## 通常運用

スケジュール実行(平日 朝7時 / 昼12時 JST)で完全自動。
通知が来たら Slack で内容を確認するだけ。

## 通知の見方

### 🟢 供給回復シグナル
最重要通知。**メーカーから受注再開・正常化のアナウンスが出た可能性**。
すぐに該当URLを開いて内容確認 → 在庫確保アクション。

### 🔴 供給制限シグナル
状況悪化の通知。新たな受注停止等が出ている可能性。

### 🟡 関連キーワード
製品名・ブランド名がヒットした更新。内容確認推奨。

## メンテナンス

### 監視対象の追加
`src/targets.yaml` を編集して push するだけ。
次回実行時から自動で反映される。

### スナップショットのリセット
特定のメーカーのスナップショットを消すと、次回実行時に
「初回」扱いになり、その時点のページが基準点になる。

```bash
rm data/snapshots/kubota_chemix.txt
git commit -am "reset snapshot: kubota_chemix"
git push
```

### 全リセット
```bash
rm -rf data/snapshots/ data/history.json
```

## よくあるトラブル

### 通知が来すぎる
1. `targets.yaml` の `selector` を絞る (body → 特定領域)
2. `keywords` を厳選する
3. 一時的にカテゴリ単位で無効化する

### 通知が来ない (本当に来ないのか確認)
1. GitHub Actions のログを確認
2. `--dry-run` で手動実行 → ログで動作確認
3. `data/snapshots/` が更新されているか確認

### サイト構造が変わって取得できない
1. ログで該当URLのHTTPステータスを確認
2. `selector` が古くなっている可能性 → 修正
3. 一時的に該当ターゲットをコメントアウト

## サイト側の対策で取得できなくなった場合

一部メーカーは Cloudflare 等の Bot 対策を入れている可能性。
その場合の対応:

1. `User-Agent` を変える
2. `requests` → `playwright` に切り替える(将来の拡張)
3. RSS が提供されていればそちらを使う
